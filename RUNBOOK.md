# Operations Runbook: LinuxOps Monitor System Alerting

This document serves as the standard operating procedure (SOP) for L1 DevOps Engineers triaging system health alerts generated by **LinuxOps Monitor**.

---

## 1. Alert Dictionary & Thresholds

| Metric | Severity | Threshold Trigger | Meaning & Potential Impact |
| :--- | :--- | :--- | :--- |
| **CPU** | `WARNING` | $\ge 80\%$ | High load. Applications may experience minor response latency. |
| **CPU** | `CRITICAL` | $\ge 90\%$ | Severe CPU exhaustion. Risks process hanging, thread pooling limits, or system freezes. |
| **MEMORY** | `WARNING` | $\ge 85\%$ | High memory allocation. Minor swapping might begin. |
| **MEMORY** | `CRITICAL` | $\ge 95\%$ | Danger of Out-Of-Memory (OOM) killer terminating critical services. |
| **DISK** | `WARNING` | $\ge 90\%$ | Storage is running low on root partition. Log rotation may fail. |
| **DISK** | `CRITICAL` | $\ge 95\%$ | Disk space nearly exhausted. Database writes or syslog logging will fail. |
| **syslog_errors** | `WARNING` | $> 5$ matches | Repeated errors in logs. Indicate failing components, service warnings, or hardware issues. |
| **syslog_errors** | `CRITICAL` | $> 20$ matches | High volume of system errors. Implies a cascading software crash or core failure. |
| **auth_failures** | `CRITICAL` | $\ge 1$ match | Authentication failure or invalid user logins detected. Potential brute-force attack. |
| **SERVICE:<name>** | `CRITICAL` | Service stopped | A critical monitored daemon (e.g., SSH, Cron) is stopped or inactive. |

## 2. L1 Incident Response Runbook

When an alert is appended to `incidents.log` or flagged in the dashboard, perform the following triage steps.

### Step 2.1: Initial Triage (Identify the Alert)
1. Read the alert line from `incidents.log`:
   `hostname | metric | severity | timestamp`
2. Run the manual health check to confirm current state:
   ```bash
   ./run_all.sh
   ```

### Step 2.2: Triage by Metric Type

#### Case A: CPU Alert (`WARNING` or `CRITICAL`)
1. **Identify resource-hogging processes:**
   ```bash
   top -b -n 1 -o %CPU | head -n 20
   # Or using ps:
   ps -eo pid,ppid,cmd,%cpu,%mem --sort=-%cpu | head -n 10
   ```
2. **Action:**
   - Note down the PID and Command.
   - Check if it is a user process or system service.
   - If a non-essential process is hung, coordinate with L2 to kill it: `kill -15 <PID>` (graceful) or `kill -9 <PID>` (forced).

#### Case B: Memory Alert (`WARNING` or `CRITICAL`)
1. **Identify high memory consumers:**
   ```bash
   top -b -n 1 -o %MEM | head -n 20
   # Or using ps:
   ps -eo pid,ppid,cmd,%cpu,%mem --sort=-%mem | head -n 10
   ```
2. **Check swap memory status:**
   ```bash
   free -h
   ```
3. **Action:**
   - Coordinate with application owner if memory leak is suspected. Do not restart major databases without approval.

#### Case C: Disk Alert (`WARNING` or `CRITICAL`)
1. **Check file system allocation:**
   ```bash
   df -h
   ```
2. **Find the largest files/directories:**
   ```bash
   sudo du -ah / | sort -rh | head -n 20
   ```
3. **Action:**
   - Check if `/var/log` is clogged. Run logrotate manually if needed: `sudo logrotate -f /etc/logrotate.conf`.
   - Safely clean temporary files: `sudo rm -rf /tmp/*` (Verify no active locks first!).

#### Case D: Auth Failures (`CRITICAL`)
1. **Examine authentication log details:**
   ```bash
   grep -E "Failed|Invalid" /var/log/auth.log | tail -n 20
   ```
2. **Action:**
   - Identify source IP address of failures.
   - If brute-force is active, block the IP using UFW/iptables:
     ```bash
     sudo ufw deny from <offending_ip> to any
     ```
   - Report the IP block to the security team immediately.

#### Case E: Service Down (`CRITICAL`)
1. **Attempt service restart:**
   ```bash
   sudo systemctl restart <service_name>
   ```
2. **Verify status:**
   ```bash
   sudo systemctl status <service_name>
   ```
3. **Action:**
   - If restart fails, inspect service logs:
     ```bash
     journalctl -u <service_name> -n 50 --no-pager
     ```

## 3. Escalation Procedures

If the issue cannot be resolved using Step 2 within **15 minutes**, escalate to the L2 DevOps team.

1. **Information to Gather Before Escalating:**
   - Alert details from `incidents.log`
   - Output of `df -h`, `free -h`, or `top` (depending on the alert type)
   - Relevant log snip from `/var/log/syslog` or `/var/log/auth.log`
   - Steps already taken (e.g., "Attempted service restart, failed with code X")
2. **Log Escalation:**
   Update the incident ticket with notes and tag the L2 on-call engineer.

---

## 4. Shift Handover Protocol

At the end of your shift, perform the following steps to ensure a clean handover:

1. **Filter incidents logged during your shift:**
   ```bash
   # Extract alerts for the last 8 hours
   grep -E "$(date -d '8 hours ago' '+%Y-%m-%d')" incidents.log
   ```
2. **Create the handover report:**
   - **Active Incidents:** Incidents still unresolved or pending L2 review.
   - **Resolved Incidents:** Incidents closed during the shift, detailing resolution (e.g. "IP blocked", "Service restarted").
   - **Maintenance:** Any scheduled downtime or deployments to monitor.
3. **Review incidents.log with the incoming engineer:**
   Ensure they sign off on active warnings.
